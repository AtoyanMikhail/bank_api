Базовая система функционирования банка на Go. Применение различных концепций Backend.  
[Backend Master Class (Golang + Postgres + Kubernetes + gRPC + AWS)](https://www.udemy.com/course/backend-master-class-golang-postgresql-kubernetes/?srsltid=AfmBOopmbz7XkcdwiUsBAn8yw9IEuk3J13_RFbv82ZjJQAOcUF-UYrHs)

## ⚡ Действия, выполненные во время проекта:
Разрабатывается базовый backend веб-сервис банка. Изучено как проектировать, разрабатывать и внедрять backend веб-сервис с нуля, предоставляя API для выполнения frontend следующих действий:
   1. Создание и управление банковскими счетами.
   2. Регистрация всех изменений баланса в каждом из счетов.
   3. Выполнение денежного перевода между 2 счетами.

## 💻 Как запустить проект локально:
1. Для запуска сервисов выполните файл ```docker-compose.yml```, который находится внутри проекта следующей командой (команда для выполнения в linux)
```bash
   sudo docker compose up -d
```
2. Тестирование HTTP endpoints:
   - Протестируйте endpoints HTTP сервиса с помощью Postman: [РУКОВОДСТВО ПО ТЕСТИРОВАНИЮ API](https://documenter.getpostman.com/view/23701330/2s9YXnyyKm#46fb489f-3fb0-4708-9ada-80047d2bc65d)
   - Также можете протестировать сервисы через ```localhost:7070/swagger/```
3. Информация об архитектуре базы данных: [ДОКУМЕНТАЦИЯ БД](https://dbdocs.io/valrichter/go_basic_bank), ПАРОЛЬ: secret

### 🗃️ Работа с БД [PostgreSQL + sqlc]
Углубился в проектирование баз данных, что позволило мне эффективно моделировать и управлять данными. Взаимодействовал с базой данных, используя транзакции и понял уровни изоляции базы данных. Кроме того, научился использовать Docker для создания локальных сред разработки и GitHub Actions для автоматизации модульных тестов.

**1.** Схема БД и связь между таблицами
   - Создать Account (Owner, Balance, Currency)
   - Регистрировать все изменения баланса в счете (Entry)
   - Делать переводы денег между 2 Accounts (Transfer)

   <img src="https://github.com/valrichter/basic-system-bank/assets/67121197/f0087f1e-ab3b-4532-a7bc-1a578c7c1e2c"/><br>

**2.** Настройка образа PostgreSQL в Docker и создание БД с помощью *.sql* файла
   - Были добавлены индексы (indexes) наиболее важных атрибутов каждой таблицы для большей эффективности при поиске

**3.** Создание версий БД. Настройка golang-migrate для выполнения миграций БД с одной версии на другую:
   - Добавлен Makefile для большего удобства при выполнении необходимых команд

<img src="https://github.com/valrichter/go-basic-bank/assets/67121197/f45876db-0fe9-4b3a-9e38-7256e346bb16"/><br>

**4.** Генерация базового CRUD для таблиц Account, Entry & Transfer с помощью sqlc. Настройка sqlc для выполнения SQL запросов с кодом Go
   - Как это работает:
      - **Вход**: Написание запроса в SQL ---> **Черный ящик**: [sqlc] ---> **Выход**: Функции на Golang с интерфейсами для их использования и выполнения запросов

**5.** Генерация ложных данных и создание Unit Tests для CRUD таблицы Account:
   - Использование файла ```random.go```

**6.** Создание транзакции ```StoreTx.go``` со свойствами ACID для перевода денег между 2 Accounts и соответствующий Unit Test
   - Бизнес-функциональность для реализации ---> Перевод с банковского счета "account1" на банковский счет "account2" 10 USD
   - Шаги реализации:
     1. Создать запись о переводе 10 USD
     2. Создать ```entry``` денег для account1 с суммой = -10
     3. Создать ```entry``` денег для account2 с суммой = +10
     4. Вычесть 10 USD из общего баланса, который имеет account1
     5. Добавить 10 к балансу account2

**7.** Создание Unit Tests (TDD) с go routines для имитации одновременных транзакций. Применение ```transaction locks``` с предложением ```FOR UPDATE``` для предотвращения чтения или записи неправильных значений одной и той же переменной.
   - Как мы используем ```transaction locks```:
     - Если transaction1 хочет получить доступ к переменной, которая в данный момент используется transaction2, transaction1 должна ждать, пока transaction2 завершится COMMIT или ROLLBACK, прежде чем сможет получить доступ к указанной переменной  
   
**8.** Изменение кода для предотвращения ситуаций deadlock и Unit Test для transactions deadlocks. Уточним, что это можно только предотвратить/смягчить и исправить в коде и бизнес-логике
   - Транзакция A для завершения нуждается в Data 2, которая используется (и следовательно заблокирована) Транзакцией B
   - В свою очередь Транзакция B для завершения нуждается в Data 1, которая используется (и следовательно заблокирована) Транзакцией A
   - Это вызывает проблему Deadlock

**10.** Реализация непрерывной интеграции (CI) с GitHub Actions для обеспечения качества кода и снижения возможных ошибок
   - ```Workflow``` состоит из нескольких Jobs
   - Каждый ```Job``` - это автоматизированный процесс
   - Jobs могут выполняться либо по ```event```, который происходит в репозитории github, либо установив ```scheduled``` или ```manually``` (вручную)
   - Для выполнения Job нам нужно указать Runner для каждого из них
   - ```Runner``` - это сервер, который прослушивает доступные Jobs и выполняет только один Job за раз. Это похоже на контейнер docker
   - Затем каждый Runner сообщает о своем прогрессе, логах и результатах в github
   - Job - это набор ```Steps```, которые будут выполняться в одном Runner
   - Все Jobs выполняются параллельно, кроме случаев, когда есть некоторые Jobs, которые зависят друг от друга, тогда они выполняются последовательно
   - ```Step``` - это отдельные задачи, которые выполняются последовательно внутри Job
   - Step может содержать одно или несколько Actions, которые выполняются последовательно
   - ```Action``` - это независимая команда, и они могут быть повторно использованы. Например: ```actions/checkout@v4```, которая проверяет, работает ли наш код локально

### 🧩 Построение RESTful HTTP JSON API [Gin + JWT + PASETO]
Разработал RESTful APIs, используя фреймворк Gin на Golang. Научился загружать конфигурации приложения, имитировать mock базы данных для надежных тестов и применять аутентификацию пользователей, обеспечивая безопасность API с помощью токенов JWT и PASETO.

**1.** Реализация базового RESTful HTTP API с фреймворком Gin, настройка сервера и добавление функций createAccount, getAccount по id и listAccount для перечисления счетов с помощью пагинации с соответствующей валидацией данных, полученных через JSON

**2.** Создание и настройка переменных окружения ```.env``` с инструментом Viper

**3.** Реализация database mock (временная БД) с GoMock для тестирования методов в ```account.go``` HTTP API и достижения 100% покрытия метода GetAccounts
   - Зачем реализовывать Mock Database:
     - Написать независимые тесты проще, потому что каждый тест будет использовать свою собственную БД
     - Более быстрые тесты, так как не нужно ждать подключения к БД и также нет ожидания выполнения запросов. Все действия выполняются в памяти
     - Позволяет писать тесты со 100% покрытием, поскольку мы можем настроить крайние случаи

**4.** Реализация ```transfer.go``` HTTP API для отправки денег между двумя счетами и добавлен ```validator.go``` для валидации валюты счетов, связанных с денежным переводом

**5.** Изменение в базе данных. 
   - Добавлена таблица User, чтобы каждый пользователь мог иметь различные Accounts с различными Currency как ARS, USD или EUR

<img src="https://github.com/valrichter/go-basic-bank/assets/67121197/54005e6b-ebad-4689-af1d-d1b602b25c9a"/><br>

**6.** Реализация и тестирование CRUD для таблицы Users, обработка ошибок PostgreSQL и исправление API ```account.go``` для работы с новой таблицей Users

**7.** Реализация API ```user.go``` и шифрование пароля Users с использованием bcrypt
   - Как это работает:
      - **Вход**: password123 ---> **Черный ящик**: bcrypt ---> **Выход**: $2a$10$BqwmET/4eq5.uIibth/rrOqSC2eqo5cy80Yj2RKuLicpRCIm1RlX. 

**8.** Создание более надежных unit test с пользовательским компаратором gomock для функции createUser API ```user.go```.

**9.** Понимание работы аутентификации токенами, различия между JWT и PASETO, проблемы безопасности JWT и как работает PASETO
   - Как работает аутентификация по токену:
     - Клиент предоставляет имя пользователя и пароль
     - Если они правильные, сервер создаст и подпишет токен с секретным ключом
     - Затем отправит токен доступа
     - Затем, если клиент желает получить доступ к какому-либо ресурсу, он делает это, используя токен в запросе
     - Сервер проверяет токен, если он действительный, авторизует запрос

**10.** Создание и проверка токенов JWT и PASETO с соответствующими тестами

**11.** Реализация API входа для возврата токена доступа как в PASETO, так и в JWT
   - Продолжительность токена входа установлена на 15 минут

**12.** Реализация middleware аутентификации и правил авторизации с использованием Gin. Позволяет более эффективно обрабатывать ошибки

**3.** Введение в gRPC, как он работает и почему лучше HTTP для связи между API
   - Remote procedure call framework (RPC):
      - Клиент может выполнить удаленную процедуру на сервере
      - Взаимодействие удаленного кода обрабатывается gRPC
      - Код API и структура данных генерируются автоматически с помощью Protocol Buffer Compiler
      - Поддерживает множественные языки программирования
   - Как это работает:
      1. Определяем RPC API и его структуру request/response, используя Protocol Buffer
      2. Генерируем gRPC stubs, генерируем код сервера и клиента на нужном языке
      3. Реализуем RPC handler на сервере
      4. Используем сгенерированный клиентский stub для вызова RPC сервера
   - 4 типа gRPC:
      - Unary gRPC
      - Client streaming gRPC
      - Server streaming gRPC
      - Bidirectional streaming gRPC
   - gRPC gateway, обработка HTTP JSON и gRPC запросов одновременно:
      - Плагин protobuf compiler
      - Генерирует proxy коды из protobuf
      - Переводит HTTP JSON вызовы в gRPC
         - In-process translation: только для unary gRPC
         - Separate proxy server: как для unary, так и для streaming gRPC
   - Написать код один раз, обслуживать gRPC и HTTP запросы
